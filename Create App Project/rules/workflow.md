# 🔄 작업 흐름 (Workflow)

> 앱 개발의 전체 프로세스와 각 단계별 규칙을 정의합니다.

---

## 📊 전체 워크플로우

```
┌─────────────────────────────────────────────────────────────────────────┐
│                                                                         │
│   1. 계획    →    2. 설계    →    3. 개발    →    4. 테스트    →    5. 배포   │
│  (Planning)     (Design)      (Development)    (Testing)      (Deploy)  │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 📌 단계별 중요도 및 우선순위

| 단계 | 중요도 | 건너뛰기 가능 | 예상 기간 | 핵심 포인트 |
|------|--------|---------------|-----------|-------------|
| 1. 계획 | 🔴 높음 | ❌ 불가 | 1~2일 | 요구사항 명확히 |
| 2. 설계 | 🔴 높음 | ❌ 불가 | 2~3일 | 코딩 전 설계 필수 |
| 3. 개발 | 🔴 높음 | ❌ 불가 | 1~2주 | 설계대로 구현 |
| 4. 테스트 | 🔴 높음 | ❌ 불가 | 2~3일 | 버그 발견 및 수정 |
| 5. 배포 | 🟡 중간 | ⭕ 가능 | 1~2일 | 내부용이면 간소화 |

> **💡 초보자 팁**: 계획과 설계는 절대 건너뛰지 마세요. 바로 코딩부터 시작하면 나중에 2배 이상 시간이 걸립니다!

## 1️⃣ 계획 (Planning)

### 목표
요구사항을 정의하고 개발 범위를 확정합니다.

### 산출물
- [ ] 기능 요구사항 목록
- [ ] 우선순위 정의
- [ ] 일정 계획

### 진행 규칙

| 규칙 | 설명 | 중요도 | 우선순위 |
|------|------|--------|----------|
| 요구사항 문서화 | 모든 요구사항은 문서로 정리 | 🔴 높음 | 1순위 |
| 우선순위 결정 | 높음/중간/낮음으로 분류 | 🔴 높음 | 1순위 |
| 범위 확정 | MVP(최소 기능)를 먼저 정의 | 🔴 높음 | 2순위 |
| 일정 산정 | 각 기능별 예상 소요 시간 산정 | 🟡 중간 | 3순위 |

### 📌 리스크 및 대처 방법

| 리스크 | 발생 가능성 | 영향도 | 대처 방법 |
|--------|-------------|--------|-----------|
| 요구사항 누락 | 🟡 중간 | 🔴 높음 | 체크리스트 활용, 이해관계자 확인 |
| 요구사항 변경 | 🔴 높음 | 🟡 중간 | MVP 우선 개발, 버전 분리 |
| 일정 과소 산정 | 🟡 중간 | 🟡 중간 | 버퍼 20% 추가, 과거 경험 참고 |
| 범위 확장 (Scope Creep) | 🟡 중간 | 🔴 높음 | MVP 명확히 정의, 추가 기능은 다음 버전 |

### 체크리스트
```
[ ] 사용자 스토리 작성 완료
[ ] 기능 목록 확정
[ ] 우선순위 결정
[ ] 일정 계획 수립
[ ] 팀 역할 분담 (해당 시)
```

## 2️⃣ 설계 (Design)

### 목표
UI/UX를 설계하고 기술 아키텍처를 확정합니다.

### 산출물
- [ ] 화면 설계서 (와이어프레임/목업)
- [ ] 데이터 구조 정의
- [ ] API 설계 문서
- [ ] 설계 검토 체크리스트 완료

### 진행 규칙

| 규칙 | 설명 | 중요도 | 우선순위 |
|------|------|--------|----------|
| 설계 우선 | 코딩 전 반드시 설계 완료 | 🔴 높음 | 1순위 |
| 검토 필수 | `design-review-checklist.md` 완료 필수 | 🔴 높음 | 1순위 |
| 승인 후 진행 | 설계 승인 후 개발 착수 | 🔴 높음 | 2순위 |
| 변경 관리 | 설계 변경 시 문서 업데이트 필수 | 🟡 중간 | 3순위 |

### 📌 리스크 및 대처 방법

| 리스크 | 발생 가능성 | 영향도 | 대처 방법 |
|--------|-------------|--------|-----------|
| 설계 없이 개발 착수 | 🟡 중간 | 🔴 높음 | 체크리스트로 설계 완료 확인 |
| UI 설계 변경 반복 | 🟡 중간 | 🟡 중간 | 와이어프레임 먼저 승인, 상세 디자인은 나중 |
| API 설계 불일치 | 🟡 중간 | 🔴 높음 | 요청/응답 형식 사전 합의, 문서화 |
| 데이터 구조 변경 | 🟢 낮음 | 🔴 높음 | 초기에 충분히 검토, 확장성 고려 |

### 의사결정 프로세스

```
설계 초안 작성
       ↓
  팀 검토 회의
       ↓
   피드백 반영
       ↓
  최종 승인 ←─ 미승인 시 재검토
       ↓
   개발 착수
```

### 체크리스트
```
[ ] 전체 화면 목록 정의
[ ] 각 화면별 레이아웃 설계
[ ] 네비게이션 플로우 정의
[ ] 데이터 스키마 정의
[ ] API 엔드포인트 정의
[ ] 설계 검토 완료
```

> **⚠️ 경고**: 설계 검토 체크리스트(`design-review-checklist.md`)를 완료하지 않고 개발을 시작하면, 나중에 큰 수정이 필요할 수 있습니다!

## 3️⃣ 개발 (Development)

### 목표
설계에 따라 기능을 구현합니다.

### 산출물
- [ ] 소스 코드
- [ ] 코드 리뷰 완료

### 진행 규칙

| 규칙 | 설명 | 중요도 | 우선순위 |
|------|------|--------|----------|
| 브랜치 생성 | 기능별 브랜치에서 개발 | 🔴 높음 | 1순위 |
| 일일 커밋 | 최소 1일 1커밋 | 🟡 중간 | 2순위 |
| 코드 리뷰 | PR 생성 후 리뷰 필수 | 🔴 높음 | 1순위 |
| 테스트 작성 | 주요 기능은 테스트 코드 작성 | 🟢 낮음 | 3순위 |

### 📌 리스크 및 대처 방법

| 리스크 | 발생 가능성 | 영향도 | 대처 방법 |
|--------|-------------|--------|-----------|
| 설계와 다르게 구현 | 🟡 중간 | 🔴 높음 | 개발 중 설계 문서 자주 참조 |
| 코드 품질 저하 | 🟡 중간 | 🟡 중간 | 코드 리뷰 필수, 컨벤션 준수 |
| 기술적 난관 발생 | 🟡 중간 | 🟡 중간 | 조기 발견 시 팀과 공유, POC 진행 |
| main 브랜치 오염 | 🟢 낮음 | 🔴 높음 | 브랜치 전략 준수, PR 필수 |

### 개발 순서

```
1. 기본 레이아웃/네비게이션 구현 ← 🔴 1순위
       ↓
2. 정적 UI 구현 (더미 데이터) ← 🔴 2순위
       ↓
3. API 연동 ← 🔴 3순위
       ↓
4. 상태 관리 구현 ← 🟡 4순위
       ↓
5. 에러 처리/예외 케이스 ← 🟡 5순위
       ↓
6. UI 다듬기/애니메이션 ← 🟢 6순위
```

> **💡 초보자 팁**: 1~3번까지 먼저 완료하세요. 이것만 되어도 앱이 동작합니다!

### 체크리스트
```
[ ] 브랜치 생성
[ ] 기능 구현
[ ] 자체 테스트
[ ] 코드 정리 (lint, format)
[ ] PR 생성
[ ] 코드 리뷰 받기
[ ] 리뷰 반영 및 머지
```

### 목표

- [ ] 테스트 결과 리포트
- [ ] 버그 목록 및 수정 내역

### 진행 규칙

| 규칙 | 설명 | 중요도 | 우선순위 |
|------|------|--------|----------|

| 디바이스 테스트 | 최소 2개 이상 기기에서 테스트 | 🔴 높음 | 1순위 |
| 에지 케이스 | 예외 상황 테스트 | 🟡 중간 | 2순위 |

### 📌 리스크 및 대처 방법

| 리스크 | 발생 가능성 | 영향도 | 대처 방법 |
|--------|-------------|--------|-----------|
| 테스트 시간 부족 | 🟡 중간 | 🔴 높음 | 개발 완료 후 충분한 테스트 기간 확보 |

| 중요 버그 미발견 후 배포 | 🟢 낮음 | 🔴 높음 | 핵심 기능 위주 집중 테스트 |
| 테스트 결과 미기록 | 🟡 중간 | 🟡 중간 | 체크리스트 활용, 버그 즉시 기록 |

### 테스트 항목

|----------|-------------|--------|----------|

| UI | 화면이 설계대로 표시되는가? | 🔴 높음 | 1순위 |
| 네트워크 | 오프라인/느린 연결에서 동작하는가? | 🟡 중간 | 2순위 |
| 입력 | 잘못된 입력에 대해 적절히 처리하는가? | 🟡 중간 | 2순위 |
| 성능 | 로딩 시간이 적절한가? | 🟢 낮음 | 3순위 |

### 체크리스트
```
[ ] 전체 기능 테스트 완료

## 5️⃣ 배포 (Deploy)

### 목표
앱을 빌드하고 사용자에게 배포합니다.

### 산출물
- [ ] APK/AAB 파일
- [ ] Play Store 등록 (선택)

### 진행 규칙

| 규칙 | 설명 | 중요도 | 우선순위 |
|------|------|--------|----------|
```

| 리스크 | 발생 가능성 | 영향도 | 대처 방법 |
|--------|-------------|--------|-----------|
| 빌드 실패 | 🟡 중간 | 🟡 중간 | 에러 메시지 확인, Expo 문서 참조 |
| 스토어 심사 반려 | 🟢 낮음 | 🟡 중간 | 정책 사전 검토, 가이드라인 준수 |
| 배포 후 치명적 버그 발견 | 🟢 낮음 | 🔴 높음 | 충분한 테스트, 핫픽스 프로세스 준비 |
| 개인정보처리방침 누락 | 🟡 중간 | 🟡 중간 | 배포 전 필수 문서 체크리스트 확인 |

### 📱 모바일 배포 특이사항 (Mobile Specific)

1. **OTA 업데이트 (Code Push)**
   - JS 코드 변경만 있는 경우 스토어 심사 없이 즉시 배포 가능
   - 명령어: `eas update --branch production --message "Quick fix"`

2. **스토어 심사 (Review)**
   - **Google Play**: 최초 심사 3~7일 소요 (이후 1~2일)
   - **App Store**: 최초 심사 1~2일 소요
   - **반려 대응**: 사유 확인 후 수정하여 재제출 (버전 넘버 증가 필수)

### 배포 순서

```
1. 코드 프리즈 (개발 중단) ← 🔴 필수
       ↓
2. 최종 테스트 ← 🔴 필수
       ↓
3. 버전 번호 업데이트 ← 🔴 필수
       ↓
4. 릴리스 빌드 ← 🔴 필수
       ↓
5. 빌드 파일 테스트 ← 🔴 필수
       ↓
6. 배포/업로드 ← 🔴 필수
       ↓
7. 배포 확인 ← 🔴 필수
```

### 체크리스트
```
[ ] 버전 번호 업데이트
[ ] 변경 로그 작성
[ ] 릴리스 빌드 생성
[ ] 빌드 테스트
[ ] 스토어 업로드 (해당 시)
[ ] 배포 완료 확인
```

> **💡 초보자 팁**: 팀 내부용이라면 APK 파일 직접 공유로 시작하세요. Play Store 등록은 나중에 해도 됩니다!

## 🔄 반복 및 유지보수

### 피드백 반영 프로세스

```
사용자 피드백 수집
        ↓
   요청 분류 및 우선순위
        ↓
   다음 버전 계획에 반영
        ↓
   1단계(계획)부터 재시작
```

### 📌 리스크 및 대처 방법

| 리스크 | 발생 가능성 | 영향도 | 대처 방법 |
|--------|-------------|--------|-----------|
| 피드백 과다로 우선순위 혼란 | 🟡 중간 | 🟡 중간 | 요청을 분류하고 영향도 기준 정렬 |
| 유지보수 시간 부족 | 🟡 중간 | 🟡 중간 | 정기 유지보수 시간 확보 |
| 기존 코드 이해 어려움 | 🟡 중간 | 🟡 중간 | 주석 및 문서화 철저히 |

### 긴급 버그 대응

```
버그 발견/보고
      ↓
  심각도 평가
      ↓
┌─────┴─────┐
│           │
긴급      일반
 ↓          ↓
핫픽스    다음 버전
  ↓      일정 포함
즉시 배포
```

### 심각도 평가 기준

| 심각도 | 설명 | 대응 | 예시 |
|--------|------|------|------|
| 🔴 긴급 | 앱 사용 불가, 데이터 손실 | 즉시 핫픽스 | 앱 크래시, 로그인 불가 |
| 🟡 중간 | 주요 기능 오류 | 1~2일 내 수정 | 검색 결과 오류, 저장 실패 |
| 🟢 낮음 | 사소한 UI 오류 | 다음 버전에 포함 | 오타, 정렬 미세 오류 |
